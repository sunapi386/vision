# Audio generation

## Overview

The audiobook is generated by `tts.py` using [Kokoro](https://github.com/hexgrad/kokoro) TTS with voice `am_michael`. The script uses per-paragraph hash-based caching so that only new or changed text is re-synthesized on subsequent runs.

## How it works

1. **Extract narration units** from MDX sources in the same order as `build.py`. Each unit is either a heading (chapter title, section heading, sub-heading) or a paragraph (body text with an `ab-{ch}-{idx}` ID).

2. **Hash each unit's text** with its voice parameters: `sha256(voice + "|" + sample_rate + "|" + pause_seconds + "|" + text)`. The first 16 hex chars become the cache filename.

3. **Check the cache** (`audio-cache/{hash}.wav`). If the file exists, load it. If not, synthesize with Kokoro and save it.

4. **Concatenate** all cached audio for the chapter in document order, with 0.6s pauses between units.

5. **Record timestamps** for paragraph units only (headings consume audio time but don't get timestamp entries). Timestamps are positional arrays matching `ab-{ch}-{idx}` IDs.

6. **Export** chapter MP3s to `audio/` and timestamp JSON to `timestamps/`.

## Cache invalidation

The hash includes voice name, sample rate, and pause duration alongside the text. Changing any of these parameters invalidates the cache for affected paragraphs. To force full regeneration, delete the `audio-cache/` directory.

## Verification

Before generating audio, `tts.py` verifies that its paragraph extraction matches `build.py` exactly. This ensures timestamp indices align with the `ab-{ch}-{idx}` IDs in the HTML. Run verification without generating audio:

```bash
python3 tts.py verify
```

Expected output:

```
  ch1: OK 20 headings + 145 paragraphs
  ch2: OK 29 headings + 159 paragraphs
  ...
All chapters verified.
```

If verification fails, the paragraph extraction in `tts.py` has diverged from `build.py` and needs to be fixed before generating audio.

## Narration scope

- **Headings** (h1 chapter titles, h2 section headings, h3+ sub-headings) are narrated for natural audiobook flow. They appear in the audio timeline between paragraphs but do not have timestamp entries.
- **Paragraphs** (`<p>` elements) are narrated and tracked with `[start, end]` timestamps.
- **Lists** with blank lines between items produce `<p>` wrappers inside `<li>`, so they are narrated and tracked like regular paragraphs.
- **Tight lists**, tables, code blocks, and other non-paragraph elements are not narrated.

## Output files

```
audio/
  part01-world-changed.mp3
  part02-the-void.mp3
  ...
  manifest.json            # chapter metadata + durations

timestamps/
  part01-world-changed.json   # [[start, end], [start, end], null, ...]
  part02-the-void.json
  ...

audio-cache/
  a1b2c3d4e5f6g7h8.wav    # cached per-paragraph WAVs (hash-named)
  ...
```

## Dependencies

- Python 3.10+
- [kokoro](https://github.com/hexgrad/kokoro) (TTS engine)
- numpy, soundfile (audio processing)
- ffmpeg (WAV to MP3 conversion)
- markdown (paragraph extraction, same version as build.py)

## Legacy

`align.py` was the previous alignment script that used OpenAI Whisper to extract segment timestamps from monolithic chapter MP3s, then `build.py` used word-count ratios to map paragraphs to those segments. This approach broke when paragraphs were added or removed (all downstream timestamps shifted). The per-paragraph approach in `tts.py` replaces both `align.py` and the old `align_paragraphs_to_timestamps()` logic.
